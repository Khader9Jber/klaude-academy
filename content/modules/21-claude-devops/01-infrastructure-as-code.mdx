## From Console Clicking to Code

Every DevOps journey starts the same way: clicking through the AWS console, creating resources manually, and praying nobody changes them. IaC fixes that, and Claude makes writing IaC feel almost as fast as clicking through the console — but with version control, reproducibility, and auditability.

## Generating Terraform From Plain English

Here is a real prompt I used when setting up infrastructure for a SaaS application:

```
Create Terraform configuration for a production-ready web application
on AWS:

Networking:
- VPC with CIDR 10.0.0.0/16
- 3 public subnets across 3 AZs (for load balancer)
- 3 private subnets across 3 AZs (for application and database)
- NAT Gateway in one public subnet for outbound traffic from private subnets
- Internet Gateway

Compute:
- ECS Fargate cluster
- ECS service running 2 tasks (min), auto-scaling to 6
- Task: 512 CPU, 1024 memory, container port 3000
- Application Load Balancer in public subnets
- Health check on /health endpoint

Database:
- RDS PostgreSQL 15, db.t3.medium
- Multi-AZ for production
- In private subnets
- Automated backups with 7-day retention
- Not publicly accessible

Security:
- ALB security group: allow 80/443 from anywhere
- ECS security group: allow 3000 from ALB only
- RDS security group: allow 5432 from ECS only
- Secrets Manager for database credentials

Use modules for VPC, ECS, and RDS. Variables for environment name,
region, and resource sizes. Outputs for ALB DNS name and RDS endpoint.
```

Claude generates a complete Terraform project. Here is the structure it typically produces:

```
terraform/
  main.tf           — provider, backend, module calls
  variables.tf      — input variables
  outputs.tf        — output values
  terraform.tfvars  — default values
  modules/
    vpc/
      main.tf
      variables.tf
      outputs.tf
    ecs/
      main.tf
      variables.tf
      outputs.tf
    rds/
      main.tf
      variables.tf
      outputs.tf
```

### The VPC Module

Claude generates clean, modular code:

```hcl
# modules/vpc/main.tf
resource "aws_vpc" "main" {
  cidr_block           = var.vpc_cidr
  enable_dns_hostnames = true
  enable_dns_support   = true

  tags = {
    Name        = "${var.environment}-vpc"
    Environment = var.environment
  }
}

resource "aws_subnet" "public" {
  count             = length(var.availability_zones)
  vpc_id            = aws_vpc.main.id
  cidr_block        = cidrsubnet(var.vpc_cidr, 8, count.index)
  availability_zone = var.availability_zones[count.index]

  map_public_ip_on_launch = true

  tags = {
    Name        = "${var.environment}-public-${count.index + 1}"
    Environment = var.environment
  }
}

resource "aws_subnet" "private" {
  count             = length(var.availability_zones)
  vpc_id            = aws_vpc.main.id
  cidr_block        = cidrsubnet(var.vpc_cidr, 8, count.index + 10)
  availability_zone = var.availability_zones[count.index]

  tags = {
    Name        = "${var.environment}-private-${count.index + 1}"
    Environment = var.environment
  }
}
```

Every resource is tagged. Every count uses the availability zones variable. The CIDR subnetting is calculated automatically. This is not boilerplate — it is production-ready configuration.

## Pulumi Alternative

If you prefer Pulumi's programming model:

```
Convert this infrastructure to Pulumi using TypeScript:
[paste Terraform config or describe the same architecture]

Use Pulumi's AWS provider. Organize into:
- src/vpc.ts — VPC, subnets, gateways
- src/ecs.ts — cluster, service, task definition
- src/rds.ts — database instance
- src/index.ts — main stack, exports

Use Pulumi's ComponentResource pattern for reusable modules.
```

Claude generates typed, IDE-friendly infrastructure code that reads like any other TypeScript project.

## State Management and Drift Detection

State issues are the bane of Terraform. Claude helps diagnose them:

```
I'm getting this error when running terraform plan:

Error: Resource already exists
  aws_ecs_cluster.main: A cluster with name "prod-cluster" already exists

But I didn't create it manually. What happened and how do I fix it?
```

Claude will explain that someone (or another Terraform workspace) likely created the resource outside your state file. It suggests:

```bash
# Import the existing resource into your state
terraform import aws_ecs_cluster.main arn:aws:ecs:us-east-1:123456789:cluster/prod-cluster

# Then run plan to see if config matches reality
terraform plan
```

For drift detection, ask Claude to help write a monitoring script:

```
Create a script that runs terraform plan in CI and posts a Slack
notification if any drift is detected. Use GitHub Actions. The
script should:
- Run terraform plan with -detailed-exitcode
- Exit code 2 means changes detected (drift)
- Parse the plan output for a summary of changed resources
- Post to Slack with the resource list and a link to the CI run
```

## Migrating From Console to IaC

This is one of Claude's most practical uses. You have resources running in AWS that were created manually. You need to bring them under Terraform management.

```
I have these resources running in AWS that I created manually:
- A VPC (vpc-0abc123) with 2 public and 2 private subnets
- An RDS PostgreSQL instance (myapp-db-prod)
- An ECS cluster (myapp-prod) with 1 service running 3 tasks
- An ALB (myapp-alb) with a target group pointing to the ECS service
- A CloudFront distribution in front of the ALB
- An S3 bucket for static assets (myapp-assets-prod)

Generate Terraform configuration that matches this existing setup.
Include terraform import commands for each resource. Group the
resources logically.
```

Claude generates both the `.tf` files and the corresponding `terraform import` commands. You run the imports, then `terraform plan` to verify the configuration matches reality. If there are differences, Claude helps reconcile them.

## Infrastructure Review Prompts

After generating or modifying Terraform, ask Claude to audit it:

```
Review this Terraform configuration for:
1. Security issues (open ports, public access, overly broad IAM)
2. Cost optimization (right-sized instances, reserved capacity options)
3. Reliability (multi-AZ, backup retention, health checks)
4. Best practices (tagging, naming conventions, module usage)
5. Missing resources (WAF, CloudWatch alarms, log groups)
```

Claude catches things like:

- RDS instance without `deletion_protection = true`
- Security group with `0.0.0.0/0` on a port that should be restricted
- ECS tasks without CloudWatch log configuration
- Missing tags that break cost allocation
- NAT Gateway in only one AZ (single point of failure)

These are the issues that survive code review because reviewers focus on logic, not infrastructure configuration details. Claude catches them consistently.

## Environment Promotion

Use variables and workspaces to manage multiple environments:

```
Refactor this Terraform to support dev, staging, and production
environments using Terraform workspaces:
- dev: smaller instances (db.t3.micro, 1 ECS task, no multi-AZ)
- staging: medium instances (db.t3.small, 2 ECS tasks, no multi-AZ)
- production: full setup (db.t3.medium, 2-6 ECS tasks, multi-AZ)

Create a tfvars file for each environment. Use workspace-based
variable selection in main.tf.
```

Claude creates `dev.tfvars`, `staging.tfvars`, and `production.tfvars` with appropriate values, and updates the configuration to use conditional logic or variable maps for environment-specific settings.

## Key Takeaway

Claude Code turns infrastructure-as-code from a specialized skill into an accessible workflow. Describe your architecture in plain English with specific resource types, networking requirements, and security constraints. Always review generated configurations before applying — infrastructure mistakes have real cost and security implications. Use Claude to generate modular, environment-aware configurations, and to audit existing infrastructure for security and best-practice compliance. The migration from console-clicking to IaC is where Claude provides the most value — it generates both the configuration and the import commands to bring existing resources under management.
