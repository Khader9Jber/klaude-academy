## Pipelines Are Tedious — Claude Makes Them Less So

CI/CD pipeline configuration is one of those tasks that feels like it should be simple but never is. Every platform has its own YAML dialect, its own caching syntax, its own way of handling secrets and environments. Claude knows all of them.

## GitHub Actions: The Full Pipeline

Here is the prompt I use for most new projects:

```
Create a GitHub Actions workflow at .github/workflows/ci.yml for a
Next.js 15 app with:

Triggers:
- Push to main (deploy to production)
- Push to develop (deploy to staging)
- Pull requests to main or develop (run checks only)

Jobs:
1. lint — ESLint check
2. typecheck — TypeScript compiler (tsc --noEmit)
3. test — Vitest unit tests with coverage report
4. build — Next.js production build
5. deploy-staging — deploy to Vercel preview (on develop push)
6. deploy-production — deploy to Vercel production (on main push)

Requirements:
- Node 20
- pnpm for package management
- Cache pnpm dependencies between runs
- Jobs 1-3 run in parallel, job 4 depends on all three passing
- Deploy jobs depend on build passing
- Upload coverage as an artifact
- Post coverage summary as a PR comment
```

Claude generates a complete workflow:

```yaml
name: CI/CD

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main, develop]

concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
        with:
          version: 9
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'pnpm'
      - run: pnpm install --frozen-lockfile
      - run: pnpm lint

  typecheck:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
        with:
          version: 9
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'pnpm'
      - run: pnpm install --frozen-lockfile
      - run: pnpm tsc --noEmit

  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
        with:
          version: 9
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'pnpm'
      - run: pnpm install --frozen-lockfile
      - run: pnpm test -- --coverage
      - uses: actions/upload-artifact@v4
        with:
          name: coverage
          path: coverage/

  build:
    needs: [lint, typecheck, test]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
        with:
          version: 9
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'pnpm'
      - run: pnpm install --frozen-lockfile
      - run: pnpm build

  deploy-staging:
    needs: [build]
    if: github.ref == 'refs/heads/develop' && github.event_name == 'push'
    runs-on: ubuntu-latest
    environment: staging
    steps:
      - uses: actions/checkout@v4
      - uses: amondnet/vercel-action@v25
        with:
          vercel-token: ${{ secrets.VERCEL_TOKEN }}
          vercel-org-id: ${{ secrets.VERCEL_ORG_ID }}
          vercel-project-id: ${{ secrets.VERCEL_PROJECT_ID }}

  deploy-production:
    needs: [build]
    if: github.ref == 'refs/heads/main' && github.event_name == 'push'
    runs-on: ubuntu-latest
    environment: production
    steps:
      - uses: actions/checkout@v4
      - uses: amondnet/vercel-action@v25
        with:
          vercel-token: ${{ secrets.VERCEL_TOKEN }}
          vercel-org-id: ${{ secrets.VERCEL_ORG_ID }}
          vercel-project-id: ${{ secrets.VERCEL_PROJECT_ID }}
          vercel-args: '--prod'
```

The `concurrency` group cancels outdated runs. The `--frozen-lockfile` ensures reproducible installs. The `environment` keyword on deploy jobs enables GitHub's environment protection rules.

## Debugging Failing Pipelines

Pipeline debugging is where Claude saves the most time. You are staring at a red X and a 200-line log. Claude cuts through it:

```
My GitHub Actions workflow is failing at the test step. Here's the error:

FAIL src/lib/__tests__/auth.test.ts
  ● AuthService › should hash password correctly
    connect ECONNREFUSED 127.0.0.1:5432

All tests pass locally. The workflow runs on ubuntu-latest with Node 20.
```

Claude immediately identifies the issue: the tests depend on a PostgreSQL database, but the workflow does not start one. It suggests adding a service container:

```yaml
test:
  runs-on: ubuntu-latest
  services:
    postgres:
      image: postgres:16
      env:
        POSTGRES_USER: test
        POSTGRES_PASSWORD: test
        POSTGRES_DB: testdb
      ports:
        - 5432:5432
      options: >-
        --health-cmd pg_isready
        --health-interval 10s
        --health-timeout 5s
        --health-retries 5
  env:
    DATABASE_URL: postgresql://test:test@localhost:5432/testdb
  steps:
    # ...existing steps...
```

### Common Pipeline Failures Claude Resolves

**"Module not found" in CI but works locally:**

Usually a case-sensitivity issue (macOS is case-insensitive, Linux is not), or a missing dependency in `package.json` that is globally installed on your machine.

**"Permission denied" on deployment scripts:**

```
Add chmod +x to scripts before running them:
- run: chmod +x ./scripts/deploy.sh && ./scripts/deploy.sh
```

**Flaky tests in CI:**

```
These tests pass locally but fail intermittently in CI. Here are the
last 5 runs: 3 passed, 2 failed. The failing tests are:
[paste test names and errors]

The failures seem to be timing-related. How do I fix this?
```

Claude suggests adding retries, increasing timeouts, or refactoring tests that depend on timing to use proper async patterns.

## Pipeline Performance Optimization

Slow pipelines kill developer productivity. Ask Claude to optimize:

```
My CI pipeline takes 12 minutes. Here's the breakdown:
- Checkout: 5s
- Install dependencies: 3m 20s
- Lint: 45s
- Typecheck: 1m 10s
- Test: 4m 30s
- Build: 2m 15s

How can I make it faster? I'm using pnpm and GitHub Actions.
```

Claude suggests:

- Aggressive dependency caching (pnpm store cache + node_modules cache)
- Running lint, typecheck, and test in parallel (separate jobs)
- Using `turbo` for cached builds if applicable
- Conditional path-based execution (skip tests if only docs changed)
- Using larger runners for build-heavy steps

## Key Takeaway

CI/CD pipeline generation with Claude eliminates the trial-and-error cycle of pipeline configuration. Specify your platform, tech stack, stage sequence, and deployment targets in one prompt. For debugging, paste the failing step's log output along with the workflow configuration — Claude identifies whether the issue is a missing service, a permissions problem, or a configuration error. Always use platform-native secret management, never hardcode credentials, and add security scanning as a standard pipeline stage. Optimize slow pipelines by caching aggressively and parallelizing independent stages.
