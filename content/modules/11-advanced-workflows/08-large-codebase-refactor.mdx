## Refactoring at Scale

Changing one file is easy. Changing five files is manageable. Changing 50+ files — renaming a core interface, migrating from one ORM to another, restructuring an entire module hierarchy — that's where most developers either give up, make a mess, or spend two weeks on what should have been two days.

Claude Code is built for this kind of work. It can read files, understand dependencies, generate changes, and verify results at a pace no human can match. But it needs a strategy. A naive "refactor everything" prompt produces chaos. A structured approach produces clean, verified, incremental changes.

## The Refactoring Framework

Every large refactor follows the same framework:

```
1. INVENTORY  → What files are affected?
2. PLAN       → What's the target state? What's the order?
3. GOLDEN     → Refactor one file perfectly as a reference
4. BATCH      → Apply the pattern in small batches
5. VERIFY     → Tests pass? Types check? Behavior unchanged?
6. COMMIT     → Save progress, create rollback points
```

Let's walk through a real example.

## Example: Migrating from Callbacks to Async/Await

You have a Node.js application with 45 service files that use callback-style database access. You want to migrate everything to async/await with Prisma.

### Step 1: Inventory

```
I need to migrate our database layer from callback-style to 
async/await with Prisma. Find all files that:
1. Import the old database module (src/lib/db-legacy.ts)
2. Use callback patterns (functions with cb or callback parameters)
3. Contain raw SQL queries

List every affected file with a count of callback usages per file.
Sort by dependency depth — files with no importers first.
```

Claude scans the codebase and produces:

```
Affected files (sorted by dependency depth):

Leaf nodes (no importers):
  src/services/AnalyticsService.ts — 3 callbacks
  src/services/NotificationService.ts — 2 callbacks
  src/services/AuditLogService.ts — 4 callbacks

Mid-level (imported by 1-3 files):
  src/services/ProductService.ts — 6 callbacks
  src/services/OrderService.ts — 8 callbacks
  src/services/CartService.ts — 5 callbacks
  ...

Core (imported by 10+ files):
  src/services/UserService.ts — 12 callbacks
  src/services/AuthService.ts — 9 callbacks
  src/lib/db-legacy.ts — the module itself

Total: 45 files, ~180 callback instances
```

Now you have a map. You know the scope and the order.

### Step 2: Plan

```
Here's the migration plan. Review it and flag any issues:

Phase 1 — Prisma Setup
  - Add Prisma schema based on existing raw SQL table definitions
  - Generate initial migration from existing database
  - Keep old db-legacy module running alongside Prisma

Phase 2 — Leaf Nodes (12 files, no importers)
  - Migrate services that nothing else depends on
  - Each file: replace callbacks with async/await, swap db-legacy for Prisma
  - Run tests after each batch of 4 files

Phase 3 — Mid-Level (20 files)
  - Same migration pattern
  - Run tests after each batch

Phase 4 — Core (8 files + db-legacy)
  - Most careful phase — these files break everything if wrong
  - One file at a time with full test suite run

Phase 5 — Cleanup
  - Remove db-legacy module
  - Remove callback utility types
  - Final full test run

Does this plan make sense? What am I missing?
```

Claude reviews the plan and might flag things like: "OrderService has a complex transaction that uses nested callbacks — that one needs special handling" or "The AuditLogService writes to a different database, make sure the Prisma connection handles that."

### Step 3: Golden File

Pick a simple leaf node and refactor it perfectly:

```
Migrate @src/services/AnalyticsService.ts from callbacks to 
async/await with Prisma. This is our golden file — get it 
exactly right because every other file will follow this pattern.

Show me:
1. The full before/after comparison
2. Every change explained
3. Run its tests to verify behavior is preserved
```

Claude produces the migration. You review it meticulously:

- Are all callbacks properly converted to async/await?
- Are Prisma queries correct (right model names, right methods)?
- Are error handling patterns preserved?
- Do the tests pass?

If anything is wrong, fix it now. This file is your reference for the next 44 files.

### Step 4: Batch Migration

With the golden file verified, migrate in batches:

```
Migrate these 4 files following EXACTLY the same pattern as 
the golden file @src/services/AnalyticsService.ts:

@src/services/NotificationService.ts
@src/services/AuditLogService.ts
@src/services/ReportService.ts
@src/services/ExportService.ts

For each file:
1. Replace callback db calls with async/await Prisma calls
2. Follow the error handling pattern from the golden file
3. Don't change the public method signatures
```

After Claude finishes the batch:

```
Run the tests for all four migrated files plus the golden file.
Show me any failures.
```

Green? Commit and move to the next batch.

```bash
git add src/services/NotificationService.ts src/services/AuditLogService.ts \
  src/services/ReportService.ts src/services/ExportService.ts
git commit -m "refactor: migrate 4 leaf services from callbacks to async/await"
```

Red? Fix the failures before moving on. Never batch-migrate without testing between batches.

### Step 5: Handle the Hard Cases

Every large refactor has files that don't fit the standard pattern:

```
The OrderService has a complex transaction that uses 3 nested 
callbacks to process an order (validate inventory → charge 
payment → update stock). This doesn't map cleanly to simple 
async/await.

Read @src/services/OrderService.ts and suggest how to handle 
the transaction. Options:
1. Prisma's $transaction with sequential operations
2. Prisma's interactive transactions
3. Something else?

Show me the approach, don't implement yet.
```

Discuss the approach. Approve it. Then implement.

### Step 6: Core Files (One at a Time)

For the most-imported files, go one at a time with a full test suite run:

```
Migrate @src/services/UserService.ts following the golden file 
pattern. This file is imported by 12 other files — be extra 
careful. Don't change any public method signatures or return types.
```

After Claude finishes:

```
Run the FULL test suite, not just UserService tests.
I want to see if anything that imports UserService broke.
```

Full green? Commit. Move to the next core file.

## Using Agent Teams for Parallel Refactoring

If you have a Max plan with agent teams, you can parallelize independent batches:

```
Use sub-agents to migrate these independent leaf services in parallel:

Agent 1: @src/services/AnalyticsService.ts and @src/services/ReportService.ts
Agent 2: @src/services/ExportService.ts and @src/services/CacheService.ts
Agent 3: @src/services/NotificationService.ts and @src/services/WebhookService.ts

All agents should follow the golden file pattern from 
@src/services/UserService.ts. Run tests after each file.
```

This cuts the migration time significantly for independent files. Never parallelize files that depend on each other.

## Verification Strategies

### Checkpoint Testing

After every batch:

```bash
# Type check
npx tsc --noEmit

# Run affected tests
npx vitest run src/services/*.test.ts

# Run full suite every 3 batches
npx vitest run
```

### Behavioral Verification

For critical services, go beyond unit tests:

```
Compare the behavior of the old and new OrderService:
1. Create a test order with the old callback version
2. Create the same test order with the new async version
3. Verify both produce identical database state

Show me the comparison.
```

### Dependency Graph Check

After migrating core files:

```
Check that no file in src/ still imports from db-legacy.ts.
If any do, list them — those are files we missed.
```

## Recovery When Things Go Wrong

### Rollback to Last Checkpoint

```bash
# If a batch goes wrong, reset to the last commit
git stash  # save current work
git log --oneline -5  # find the last good commit
# Continue from the last good commit
```

### Partial Rollback

If only some files in a batch are broken:

```
The migration of PaymentService introduced a bug — the webhook 
handler is failing. Revert ONLY the PaymentService changes while 
keeping the other 3 files in this batch.
```

### Fallback: Coexistence

If the refactor is taking too long, support both old and new patterns temporarily:

```
Create an adapter that makes db-legacy calls look like Prisma calls.
This way, unmigrated files still work while we finish the migration 
at a sustainable pace.
```

## Key Takeaway

Large codebase refactoring with Claude follows a disciplined framework: inventory affected files, plan the order (leaf nodes first), create a golden reference file, batch changes in groups of 3-5 with testing between batches, and handle edge cases individually. Commit after every successful batch to create rollback points. For core files imported by many others, refactor one at a time with full test suite runs. Agent teams can parallelize independent batches. The critical discipline is never skipping verification — type checks and tests between every batch catch issues when they're cheap to fix, not after they've cascaded through 40 files.
