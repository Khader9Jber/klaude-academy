## The Strategic Question

You have a codebase with 20% test coverage. Your team lead says: "We need to get to 85% coverage in two weeks." Where do you even start?

This is where Claude transforms from a code assistant into a strategic partner. Instead of writing random tests to hit a coverage number, Claude can analyze your entire codebase and build a testing roadmap that maximizes the value of every test you write.

## Step 1: Codebase Analysis

```bash
claude "Analyze this codebase and produce a test strategy assessment.

Read the project structure and key files. For each module, assess:
1. Current test coverage (check for existing test files)
2. Business criticality (how bad is it if this breaks?)
3. Complexity (cyclomatic complexity, number of branches)
4. Dependencies (what other modules depend on this?)
5. Change frequency (how often is this code modified?)

Modules to analyze:
@src/lib/auth.ts
@src/lib/payments.ts
@src/lib/database.ts
@src/lib/email.ts
@src/lib/cache.ts
@src/lib/validators.ts
@src/app/api/

Output a risk matrix:

| Module | Coverage | Risk | Complexity | Priority |
|--------|----------|------|------------|----------|

Then list the top 10 functions that need tests most urgently."
```

## Step 2: The Test Pyramid for Your Project

```bash
claude "Based on the codebase analysis, design a test pyramid 
specifically for this project.

For each layer, specify:
- What to test
- How many tests (approximate)
- Which framework to use
- Expected execution time

Layer 1: Unit Tests (base)
Layer 2: Integration Tests (middle)
Layer 3: E2E Tests (top)

Include the ratio: for every 1 E2E test, how many integration 
and unit tests should we have?

Consider our specific tech stack: Next.js, Supabase, Stripe."
```

Claude designs a project-specific pyramid:

```
## Test Pyramid for This Project

### Layer 1: Unit Tests (70% of test effort)
Framework: Vitest
Target: ~200 tests
Execution time: < 30 seconds

What to test:
- All pure functions in src/lib/ (validators, formatters, helpers)
- Business logic in payment and auth modules (with mocked deps)
- React hooks (with renderHook)
- State management logic (Zustand stores)
- Type guards and data transformers

### Layer 2: Integration Tests (20% of test effort)
Framework: Vitest + Supertest
Target: ~60 tests
Execution time: < 3 minutes

What to test:
- API routes with real database (test DB)
- Authentication flow end-to-end (login → session → protected route)
- Payment flow with Stripe test mode
- Database queries and transactions
- Email sending (mock SMTP, verify content)

### Layer 3: E2E Tests (10% of test effort)
Framework: Playwright
Target: ~25 tests
Execution time: < 8 minutes

What to test:
- Critical user journeys:
  1. Sign up → verify email → first login
  2. Create project → add tasks → invite member
  3. Checkout → payment → confirmation
  4. Settings → change password → re-login
- Cross-browser smoke tests (Chrome, Firefox)
- Mobile responsive critical flows

### Ratio
For every 1 E2E test: 2-3 integration tests : 8-10 unit tests
```

## Step 4: Executing the Strategy

Use Claude to execute each day's plan:

```bash
# Day 1: Auth module
claude -p "Generate the Day 1 tests from our test strategy.

Write Vitest tests for @src/lib/auth.ts:

1. verifyToken (8 tests):
   - Valid token returns user
   - Expired token throws AuthError
   - Malformed token throws AuthError
   - Missing token throws AuthError
   - Token with wrong secret throws AuthError
   - Token for deleted user throws AuthError
   - Token with invalid payload throws AuthError
   - Token verification calls database once (no N+1)

2. refreshSession (6 tests):
   - Valid refresh token returns new session
   - Expired refresh token is rejected
   - Used refresh token is rejected (rotation)
   - Refresh extends session by correct duration
   - Old tokens are invalidated after refresh
   - Concurrent refresh requests don't duplicate sessions

Mock: database, jwt.verify, Date.now
Follow test conventions from CLAUDE.md"
```

## Risk-Based Testing in Practice

Not everything deserves 90% coverage. Use risk levels to allocate effort:

```bash
claude "Apply risk-based testing to our codebase.

Categorize every module into risk tiers:

Tier 1 — CRITICAL (target: 90%+ coverage):
- Anything handling money, auth, or user data
- Identify these from @src/lib/ and @src/app/api/

Tier 2 — HIGH (target: 80%+ coverage):
- Core business logic, data processing
- Database queries with complex joins
- State management for critical features

Tier 3 — MEDIUM (target: 60%+ coverage):
- UI components with business logic
- Utility functions
- Configuration and setup

Tier 4 — LOW (target: 40%+ coverage):
- Static pages, about/marketing
- Admin-only features with low usage
- Deprecated code pending removal

List specific files in each tier with their current coverage 
and the tests needed to reach the target."
```

## The Ongoing Strategy

A test strategy is not a one-time exercise:

```bash
claude "Create a sustainable testing process for our team.

Add to CLAUDE.md:

## Testing Rules
1. Every PR must include tests for new/changed code
2. New features: unit tests + at least 1 integration test
3. Bug fixes: must include a regression test
4. Coverage must not decrease on any PR
5. Critical module changes: require E2E test update

## Test Review Checklist
- [ ] Tests cover happy path
- [ ] Tests cover error paths
- [ ] Tests cover edge cases
- [ ] No flaky patterns (no sleeps, no timing deps)
- [ ] Test names describe the scenario
- [ ] Mocks are realistic
- [ ] No test data leaks between tests"
```

## Key Takeaway

Going from 20% to 85% coverage is not about writing random tests until the number goes up. It is about strategy: analyze your codebase for risk and complexity, prioritize critical paths first, follow the test pyramid for the right distribution of test types, and execute in daily phases with measurable progress. Claude can analyze your codebase, design the strategy, produce the day-by-day roadmap, and then generate the actual tests for each day's plan. The combination of strategic planning and Claude's test generation speed is how you go from "we don't have enough tests" to "our test suite catches real bugs" in two weeks.
