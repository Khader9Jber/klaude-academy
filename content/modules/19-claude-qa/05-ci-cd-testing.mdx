## The Pipeline That Catches Everything

A fast test suite that nobody runs in CI is worthless. A CI pipeline that takes 45 minutes to run gets ignored. The goal is a pipeline that is fast enough that developers actually wait for it and thorough enough that it catches real bugs.

## Building the Test Pipeline with Claude

```bash
claude -p "Create a comprehensive GitHub Actions CI/CD pipeline 
for a Next.js + Supabase project.

Create .github/workflows/ci.yml with these jobs:

1. lint-and-typecheck (2 min):
   - ESLint
   - TypeScript type checking
   - Prettier check

2. unit-tests (3 min):
   - Run Vitest with coverage
   - Fail if coverage below 80%
   - Upload coverage report as artifact

3. integration-tests (5 min):
   - Start test database (Supabase local)
   - Run integration tests
   - Tear down database

4. e2e-tests (8 min):
   - Build the Next.js app
   - Start the server
   - Run Playwright tests (sharded across 3 workers)
   - Upload failed test screenshots as artifacts

5. security-scan (2 min):
   - npm audit (warn on moderate, fail on high/critical)
   - Check for secrets in code (using truffleHog or similar)

Job dependencies:
- lint-and-typecheck runs first
- unit-tests and integration-tests run in parallel after lint
- e2e-tests runs after unit and integration pass
- security-scan runs in parallel with everything

Triggers:
- Push to main: full pipeline
- Pull request: full pipeline
- Schedule: nightly at 2 AM (run with retries for flaky detection)

Cache:
- node_modules (keyed on package-lock.json hash)
- Next.js build cache
- Playwright browsers

Reference: @package.json for scripts and dependencies"
```

## The Generated Pipeline

```yaml
# .github/workflows/ci.yml
name: CI Pipeline

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]
  schedule:
    - cron: '0 2 * * *'

concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true

jobs:
  lint-and-typecheck:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm'
      - run: npm ci
      - run: npm run lint
      - run: npx tsc --noEmit
      - run: npx prettier --check "src/**/*.{ts,tsx}"

  unit-tests:
    needs: lint-and-typecheck
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm'
      - run: npm ci
      - run: npm run test:coverage
      - name: Check coverage threshold
        run: |
          COVERAGE=$(cat coverage/coverage-summary.json | jq '.total.lines.pct')
          if (( $(echo "$COVERAGE < 80" | bc -l) )); then
            echo "Coverage $COVERAGE% is below 80% threshold"
            exit 1
          fi
      - uses: actions/upload-artifact@v4
        if: always()
        with:
          name: coverage-report
          path: coverage/

  integration-tests:
    needs: lint-and-typecheck
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:15
        env:
          POSTGRES_PASSWORD: testpass
          POSTGRES_DB: testdb
        ports:
          - 5432:5432
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm'
      - run: npm ci
      - name: Run migrations
        run: npx tsx scripts/migrate.ts
        env:
          DATABASE_URL: postgresql://postgres:testpass@localhost:5432/testdb
      - name: Run integration tests
        run: npm run test:integration
        env:
          DATABASE_URL: postgresql://postgres:testpass@localhost:5432/testdb

  e2e-tests:
    needs: [unit-tests, integration-tests]
    runs-on: ubuntu-latest
    strategy:
      matrix:
        shard: [1, 2, 3]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm'
      - run: npm ci
      - name: Install Playwright browsers
        run: npx playwright install --with-deps chromium
      - name: Build app
        run: npm run build
      - name: Run E2E tests (shard ${{ matrix.shard }}/3)
        run: npx playwright test --shard=${{ matrix.shard }}/3
      - uses: actions/upload-artifact@v4
        if: failure()
        with:
          name: playwright-results-shard-${{ matrix.shard }}
          path: test-results/

  security-scan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm'
      - run: npm ci
      - name: Audit dependencies
        run: npm audit --audit-level=high
      - name: Check for secrets
        uses: trufflesecurity/trufflehog@main
        with:
          path: ./
          base: ${{ github.event.pull_request.base.sha }}
          head: ${{ github.event.pull_request.head.sha }}
```

## Flaky Test Detection and Quarantine

```bash
claude -p "Add a flaky test detection system to our CI pipeline.

Create:
1. scripts/detect-flaky.ts:
   - Parse test results JSON from Playwright/Vitest
   - Identify tests that failed then passed on retry
   - Write flaky test list to flaky-tests.json

2. .github/workflows/flaky-detection.yml:
   - Runs nightly
   - Executes the full test suite 3 times
   - Compares results across runs
   - Any test that doesn't produce the same result all 3 times is flaky
   - Creates a GitHub issue listing newly detected flaky tests

3. src/test/quarantine.ts:
   - Reads flaky-tests.json
   - Exports a list of quarantined test names
   - In CI: quarantined tests run but don't fail the build
   - Locally: quarantined tests are skipped with a warning

4. Add to CLAUDE.md:
   'When generating tests, avoid common flakiness sources:
    - Don't depend on timing (use waitFor instead of sleep)
    - Don't depend on execution order
    - Don't share state between tests
    - Always clean up test data in afterEach'"
```

## Key Takeaway

A good CI/CD test pipeline is fast enough that developers wait for it and thorough enough that it catches real bugs before they reach production. Use sharding for parallel test execution to keep E2E tests under 10 minutes. Implement quality gates that enforce coverage thresholds, zero lint errors, and clean security scans. Detect and quarantine flaky tests instead of ignoring them. Claude can generate the entire pipeline infrastructure — GitHub Actions workflows, quality gates, flaky detection, and health monitoring — from a single detailed prompt. The investment in CI/CD test infrastructure pays off with every deployment that does not break production.
