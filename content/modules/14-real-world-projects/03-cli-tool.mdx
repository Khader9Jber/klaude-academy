## What We Are Building

A project scaffolding CLI called `quickstart` that initializes new projects from templates. It demonstrates every CLI pattern: commands with subcommands, interactive prompts, progress indicators, file system operations, configuration files, and error handling.

```bash
# What the finished CLI looks like
quickstart init my-app --template express-ts
quickstart templates list
quickstart templates add my-template ./path/to/template
quickstart config set defaultTemplate express-ts
quickstart config get defaultTemplate
```

## Phase 1: CLI Skeleton

### The Setup Prompt

```
"Create a CLI tool called 'quickstart' using:
- TypeScript
- Commander.js for command parsing
- Chalk for colored output
- Ora for spinners
- Inquirer for interactive prompts
- fs-extra for file operations

Directory structure:
src/
├── index.ts          — Entry point, register all commands
├── commands/         — One file per command
├── lib/
│   ├── output.ts     — Colored output helpers (success, error, warn, info)
│   ├── config.ts     — Read/write ~/.quickstart/config.json
│   └── templates.ts  — Template registry and operations
└── types/            — TypeScript interfaces

Setup:
1. package.json with bin field pointing to dist/index.js
2. tsconfig.json targeting ES2022
3. Shebang (#!/usr/bin/env node) in the entry point
4. Create the output helpers module with these functions:
   - success(message) — green checkmark + message
   - error(message) — red X + message
   - warn(message) — yellow warning + message
   - info(message) — blue info + message
   - spinner(message) — returns an ora spinner instance
   - table(headers, rows) — formatted table output

Do NOT create any commands yet. Just the skeleton and output module."
```

### Verify It Works

```
"Build the project (npx tsc) and run: node dist/index.js --help
Show me the output."
```

Expected:

```
Usage: quickstart [options] [command]

Project scaffolding CLI

Options:
  -V, --version  output the version number
  -h, --help     display help for command
```

## Phase 2: First Command — init

```
"Create the 'init' command in src/commands/init.ts.

Usage: quickstart init <project-name> [options]
Options:
  --template <name>   Template to use (default: 'basic')
  --skip-install      Skip npm install
  --verbose           Show detailed output

Behavior:
1. Check if directory already exists → error if it does
2. If no --template flag, use Inquirer to prompt:
   - Select template from available templates
   - Enter project description
   - Select package manager (npm, pnpm, yarn)
3. Create project directory
4. Copy template files (from a templates/ directory in the CLI package)
5. Replace placeholders in template files:
   - {{PROJECT_NAME}} → project name
   - {{DESCRIPTION}} → description
   - {{YEAR}} → current year
6. Run package manager install (unless --skip-install)
7. Show success message with next steps

Use the output helpers for all messages.
Use a spinner during file copying and npm install.

For now, create one built-in template called 'basic' with:
- package.json (with placeholders)
- tsconfig.json
- src/index.ts
- .gitignore
- README.md (with placeholders)

Register the command in src/index.ts."
```

## Phase 3: Template Management

```
"Create the 'templates' command group with subcommands:

quickstart templates list
  - Show table of available templates (name, description, files count)
  - Include built-in and custom templates

quickstart templates add <name> <path>
  - Copy files from path into ~/.quickstart/templates/<name>/
  - Create template metadata (name, description, files list)

quickstart templates remove <name>
  - Cannot remove built-in templates
  - Confirm before removing custom templates (Inquirer confirm)

quickstart templates info <name>
  - Show template details: name, description, file tree, 
    placeholders used, created date

Store custom templates in ~/.quickstart/templates/.
Store the template registry in ~/.quickstart/config.json."
```

## Phase 4: Configuration

```
"Create the 'config' command group:

quickstart config set <key> <value>
  - Set a config value in ~/.quickstart/config.json
  - Valid keys: defaultTemplate, packageManager, author, license
  - Validate values (e.g., packageManager must be npm/pnpm/yarn)

quickstart config get <key>
  - Print the current value
  - If not set, show the default value

quickstart config list
  - Show all config values as a table

quickstart config reset
  - Reset all config to defaults (confirm first)"
```

## Phase 5: Error Handling

```
"Add comprehensive error handling to all commands:

1. Create an error handler wrapper in src/lib/errors.ts:
   - CLIError class with message, suggestion, and exitCode
   - Wrap every command action in a try/catch
   - On CLIError: show message + suggestion in formatted output
   - On unknown error: show 'An unexpected error occurred' 
     + 'Run with --verbose for details'
   - On --verbose: show the full stack trace

2. Add these specific error handlers:
   - Directory already exists: suggest a different name or --force
   - Template not found: list available templates
   - Permission denied: suggest running with sudo or changing permissions
   - Network error (npm install): suggest --skip-install
   - Invalid config key: list valid keys

3. Exit codes:
   - 0: success
   - 1: general error
   - 2: usage error (invalid arguments)
   - 3: permission error"
```

## Key Takeaway

Building a CLI with Claude Code follows the same pattern-first approach as API development. Start with the framework skeleton and output formatting module — these establish the conventions for every command. Build one complete command (init) with all patterns: argument parsing, interactive prompts, file operations, spinners, and error handling. Then replicate the pattern for additional commands. The output module is the CLI equivalent of a design system — it ensures every command looks and feels consistent. Error handling deserves its own phase because good CLIs are defined by how they fail, not just how they succeed.
