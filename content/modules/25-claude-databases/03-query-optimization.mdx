## The Query Whisperer

A 30-second query isn't a database problem — it's a conversation with Claude waiting to happen. Paste the query, paste the EXPLAIN output, and Claude tells you exactly why it's slow and how to fix it.

## The EXPLAIN Workflow

Here's how I optimize queries with Claude. It's the same process every time, and it works on everything from simple lookups to 200-line analytics monsters.

### Step 1: Get the EXPLAIN Output

```sql
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT o.id, o.total_amount, o.created_at,
       c.name as customer_name,
       c.email,
       array_agg(oi.product_name) as products
FROM orders o
JOIN customers c ON o.customer_id = c.id
JOIN order_items oi ON o.id = oi.order_id
WHERE o.created_at >= '2024-01-01'
  AND o.created_at < '2024-04-01'
  AND c.region = 'europe'
  AND o.status = 'completed'
GROUP BY o.id, o.total_amount, o.created_at,
         c.name, c.email
ORDER BY o.total_amount DESC
LIMIT 100;
```

### Step 3: Claude's Analysis

```
The query takes 28 seconds. Here's why and how to fix it:

PROBLEM 1: Full table scan on order_items (18 seconds)
The Seq Scan on order_items reads all 3.5M rows, but only
186K match the join condition. This is 95% wasted I/O.

FIX: The FK index on order_id already exists, but PostgreSQL
chose a Seq Scan because it estimated the join would touch a
large fraction of the table. The real problem is that order_items
is being joined before filtering.

PROBLEM 2: Full table scan on orders (no index on date + status)
The Seq Scan on orders reads all 1M rows and filters down to
234K. Adding an index on the filter conditions would let
PostgreSQL start with a much smaller set.

PROBLEM 3: No index on customers.region
Scanning 100K customers to find 45K in Europe — the filter
isn't very selective, but it's still worth indexing because it's
combined with the join.

RECOMMENDED INDEXES:

-- Index 1: Covers the WHERE clause on orders
CREATE INDEX idx_orders_date_status
    ON orders(created_at, status)
    WHERE status = 'completed';
-- Partial index: only includes completed orders,
-- making it smaller and faster

-- Index 2: Composite on customers for the region filter
CREATE INDEX idx_customers_region
    ON customers(region);

After adding these indexes, the planner should:
1. Use idx_orders_date_status to get the 62K matching orders
2. Join to customers using the region index
3. Join to order_items using the existing FK index
4. Estimated improvement: 28s → 50-100ms

QUERY REWRITE (optional further optimization):
```

```sql
-- Use a CTE to materialize the filtered orders first,
-- then join to order_items only for matching orders
WITH filtered_orders AS (
    SELECT o.id, o.total_amount, o.created_at,
           c.name as customer_name, c.email
    FROM orders o
    JOIN customers c ON o.customer_id = c.id
    WHERE o.created_at >= '2024-01-01'
      AND o.created_at < '2024-04-01'
      AND c.region = 'europe'
      AND o.status = 'completed'
)
SELECT fo.*,
       array_agg(oi.product_name) as products
FROM filtered_orders fo
JOIN order_items oi ON fo.id = oi.order_id
GROUP BY fo.id, fo.total_amount, fo.created_at,
         fo.customer_name, fo.email
ORDER BY fo.total_amount DESC
LIMIT 100;
```

That analysis found the root causes, recommended specific indexes (including a partial index), and rewrote the query for better plan selection. From 28 seconds to under 100ms.

## N+1 Query Detection

The silent killer of web app performance. Claude finds them in your code:

```bash
claude "Review this code for N+1 query problems:

async function getOrdersPage(userId: string) {
  const orders = await db.query(
    'SELECT * FROM orders WHERE user_id = $1 LIMIT 20',
    [userId]
  );

  for (const order of orders) {
    order.items = await db.query(
      'SELECT * FROM order_items WHERE order_id = $1',
      [order.id]
    );
    order.customer = await db.query(
      'SELECT * FROM customers WHERE id = $1',
      [order.customer_id]
    );
  }

  return orders;
}"
```

Claude identifies the problem and generates the fix:

```
N+1 detected: 1 query for orders + 20 queries for items +
20 queries for customers = 41 queries. Should be 1 query.

Fix:
```

```typescript
async function getOrdersPage(userId: string) {
  const orders = await db.query(`
    SELECT
      o.*,
      c.name as customer_name,
      c.email as customer_email,
      json_agg(json_build_object(
        'id', oi.id,
        'product_name', oi.product_name,
        'quantity', oi.quantity,
        'price', oi.price
      )) as items
    FROM orders o
    JOIN customers c ON o.customer_id = c.id
    LEFT JOIN order_items oi ON o.id = oi.order_id
    WHERE o.user_id = $1
    GROUP BY o.id, c.name, c.email
    ORDER BY o.created_at DESC
    LIMIT 20
  `, [userId]);

  return orders;
}
// 41 queries → 1 query
```

### ORM-Specific N+1 Fixes

```bash
claude "I'm using Prisma and I suspect N+1 queries.
Here's my resolver:

const user = await prisma.user.findUnique({
  where: { id: userId },
});
const posts = await prisma.post.findMany({
  where: { authorId: userId },
});
for (const post of posts) {
  post.comments = await prisma.comment.findMany({
    where: { postId: post.id },
  });
}

Fix the N+1 using Prisma's include/select."
```

```typescript
const user = await prisma.user.findUnique({
  where: { id: userId },
  include: {
    posts: {
      include: {
        comments: true,
      },
      orderBy: { createdAt: 'desc' },
    },
  },
});
// Prisma generates 3 optimized queries internally:
// 1. SELECT * FROM users WHERE id = $1
// 2. SELECT * FROM posts WHERE author_id = $1
// 3. SELECT * FROM comments WHERE post_id IN ($1, $2, ...)
```

## Slow Query Log Analysis

Feed Claude your slow query log and it prioritizes fixes:

```bash
claude "Here are my top 10 slowest queries from
pg_stat_statements, ordered by total_exec_time:

1. SELECT ... FROM orders WHERE user_id = $1 AND status = $2
   calls: 450,000 | mean_time: 12ms | total_time: 5400s

2. SELECT count(*) FROM events WHERE created_at > $1
   calls: 1,200 | mean_time: 3200ms | total_time: 3840s

3. SELECT ... FROM products WHERE to_tsvector(...) @@ $1
   calls: 89,000 | mean_time: 35ms | total_time: 3115s

[... more queries ...]

For each query, recommend an index or optimization.
Prioritize by total_time impact."
```

Claude prioritizes:

```
Priority 1: Query #2 — count(*) on events
  3200ms mean time is extreme. Likely missing index on created_at.
  FIX: CREATE INDEX idx_events_created ON events(created_at);
  Expected: 3200ms → 5ms (640x improvement)
  Impact: saves ~3840 seconds of DB time per stats period

Priority 2: Query #1 — orders by user + status
  12ms isn't terrible but 450K calls makes it your highest total.
  FIX: CREATE INDEX idx_orders_user_status ON orders(user_id, status);
  Expected: 12ms → 0.5ms
  Impact: saves ~5175 seconds

Priority 3: Query #3 — full-text search
  35ms per call is okay, but a GIN index would cut it to 2-5ms.
  FIX: CREATE INDEX idx_products_fts ON products
       USING GIN(to_tsvector('english', name || ' ' || description));
  Expected: 35ms → 3ms
  Impact: saves ~2848 seconds
```

## Advanced Optimization Patterns

```bash
claude "This analytics query aggregates 50M rows and takes
45 seconds. It runs every time someone opens the dashboard.
The data only changes hourly.

What are my options?"
```

Claude suggests:
1. **Materialized view** refreshed hourly
2. **Summary table** updated by a trigger on inserts
3. **Application-level caching** with 1-hour TTL
4. **Pre-computed partitions** by date range

And generates the implementation for whichever approach fits:

```sql
-- Materialized view approach
CREATE MATERIALIZED VIEW dashboard_stats AS
SELECT
    date_trunc('day', created_at) as day,
    status,
    count(*) as order_count,
    sum(total_amount) as revenue,
    avg(total_amount) as avg_order_value
FROM orders
GROUP BY 1, 2;

CREATE UNIQUE INDEX idx_dashboard_stats
    ON dashboard_stats(day, status);

-- Refresh (run this on a schedule)
REFRESH MATERIALIZED VIEW CONCURRENTLY dashboard_stats;
-- CONCURRENTLY means the old data is still readable during refresh
```

## Key Takeaway

Query optimization with Claude follows a consistent workflow: paste the query, paste the EXPLAIN ANALYZE output, provide table sizes and existing indexes. Claude reads the execution plan like a senior DBA — identifying sequential scans that should be index scans, joins operating on unnecessarily large datasets, and missing indexes for your WHERE and ORDER BY clauses. The biggest wins come from N+1 detection (41 queries to 1), adding composite indexes that match your actual WHERE clauses, and restructuring analytics queries with materialized views. Always use EXPLAIN ANALYZE (not just EXPLAIN) so Claude sees actual row counts and timings, not just estimates.
